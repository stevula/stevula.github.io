The Specificity Wars

24 September, 2015

My first exposure to HTML was in a high school web design class around 2002. We didn't use any of this fancy-schmancy CSS magic! All of our styling was in the HTML and we positioned everything by putting it inside a giant table with invisible borders. And we called that giant table a webpage. You may already be acquainted with this style of web design from the Berkshire Hathaway website (glorious!).

I took another jab at a web design class while attending university in 2011. I'm not gonna lie to you, but CSS blew my mind a bit. Everyone was really into divs back then. Divs inside of divs, inside of other divs! It was like div-ception.

So here I am, four years later and once again, everything has changed! They took the role of the old divs and split it up into all these other semantic tags, like <article>, <section>, <main>... which I tend to think of simply as more specific kinds of divs. Naturally, being the borderline perfectionist that I am, I had to figure out the best and proper way of using each of these new tags. "What are the best practices?" I asked. Well, let me tell you, there is quite the free-for-all going on right now in the world of CSS, it seems.

Some people advocate doing away with id selectors (except as JS hooks). Proponents of Atomic advocate using only one declaration per CSS block - the result of which is a strange hybrid of the inline HTML styling of the past and the website-wide mutability that CSS brings. And then there's BEM, OOCSS, SMACSS... the list goes on.

Well, the prompt for this blog post was supposed to be: What are the best practices associated with using classes vs ids? Traditionally, a class is used for a bunch of things that you want to style the same and an id for something that is styled uniquely. It used to be that simple, at least.

But now some people say you should never use an id, because maybe you will want to style other things like it later. Why not use a class instead; even if it only applies to one thing right now, you never know if you might want to use that styling on other elements later. And let's not forget that id's add complexity to the specificity structure of the CSS. And yet, others would argue that id's fulfill a purpose and we should use them for that!

Anyways, I'm not sure that was what the writers of this prompt at had in mind, but I found the debate interesting. I think I'll refrain from picking a side for the time being. The most important thing, I think, is to do things consistently within the context of the project.
