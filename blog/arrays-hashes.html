Arrays and Hashes

29 September, 2015

Two data structures, both alike in dignity... but not without their individual merits.

An Array is basically a list of items, referred to individually as "elements", which (in Ruby) are separated by commas and enclosed in square-brackets:

[element1, element2, element3]

The Ruby documentation says that an Array can contain a collection of any object, which means the elements can be strings, numerics, Booleans, other arrays... almost anything. That might not hold true on things that are sort of object-y, like Methods and Procs, but it would be interesting to test that out. An Array can also be empty:

[]

Each element in the Array is automatically assigned an "index", based on its position in the Array. The first element has an index of 0, the second has an index of 1, and so on. You can retrieve the stored value by referencing the corresponding index of the Array. For example, my_array[1] would retrieve the second element of my_array.

I should add that you can also index from the back of the Array by using negative numbers: -1 is the last element, -2 is the second to last, etc.

So, there you have it, Arrays are useful for storing pieces of information when a numeric-indexing system suits your purposes. Because of that, they're very useful when you don't know and/or care where each element is going to be in the list, like when you just want to iterate over all of the items. Now, on to Hashes!

Hashes are similar to Arrays, in that they store lists of data. The main difference is that Hashes store information as a "key-value pair", meaning that instead of each value having a numeric index, it has a unique key. The key can be a string, number, or (most often) a Symbol.

What we call a Hash in Ruby is named differently in some languages. It's called an Object in JavaScript (that's not confusing at all!) and a Dictionary in Python. In Ruby, a Hash is composed of a list of key-value pairs enclosed in angle-brackets, with keys and pairs separated by "=>" or ":", depending on the notation; each pair is separated from the next pair by a comma:

{ key1 => value1, key2 => value2 }

The advantage of Hashes is that you can retrieve a value just by referencing the key, without knowing where it is stored in the Hash. That is ideal for when you have linked data, like item/price, business/address, or questions/answers, conditions/truthiness:

  • hamburger => 6.00
  • Dev_Bootcamp => "633 Folsom, San Francisco, CA 94107"
  • favorite_color => "puce"
  • alive => true

You could create a Hash called "Steven" (that's me) and use it to store various tawdry details about me: { hair_color => "brown", felony_convictions => "unknown", blogging_level => "novice" }. You could use one to store account details on a customer, menu items with costs, or a list of all your old high school rivals and their current happiness ratings. The possibilities are endless!

If it's not clear why a key is better than a numeric index in some situations, imagine trying to look up the definition of "mercurial" in the dictionary if all the headwords were replaced with numbers. That's what an Array-based dictionary would be. I would imagine Arrays tend be to be a poor choice for large datasets, if you plan to reference specific indices regularly.

To wrap things up, Arrays excel at storing information when you have single pieces of data or if the indexing scheme is irrelevant to your current purpose. Hashes are good for storing linked data and provide a human-friendly way to access values, which is agnostic as to sequential position in the list.
