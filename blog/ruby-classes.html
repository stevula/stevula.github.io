Ruby Classes

14 October, 2015

Classes in Ruby, as the name suggests, are categories of objects. Ruby comes with a lot of built-in classes to provide us with useful categories for things such as text, numbers, and data structures. To name a few Ruby classes:

Numeric Integer Float String Array

Classes can also be subclasses of other classes. For example, Integer and Float are subclasses of Numeric.

Besides categorizing objects, classes also define the "properties" and "abilities" of objects. Classes may contain these in their definition or "inherit" them from a parent (or grandparent, or great-grandparent...) class. These "properties", or variables, and "abilities", or methods, are shared by objects of the same class.

Because classes give objects properties and abilities, they allow you to create objects that simulate real-world things or concepts. You can define a class that simulates a shopping list, a cash register, an ATM, or even a person. It is said that some have even created the game BINGO with classes, but surely such rumors are madness...

Let's say we want to make a class called CrazyRobot (Ruby convention dictates that classes be named in CamelCase). First we define the class with a name:

class CrazyRobot end

Now we have a class, but it doesn't really do anything yet. We want to make sure that new instance objects of this class can take a "name" and "material" property as arguments and assign them to variables. For that, we use the Class#initialize method:

class CrazyRobot   def initialize(name, material)     name = name     material = material   end end

Those variable assignments are actually redundant at the moment, but I just want to demonstrate something with them a bit later. Anyways, now we can create our first instance of CrazyRobot!

bender = CrazyRobot.new("Bender Bending Rodriguez", "dolomite (40%)")

This passes the values in parentheses to the #initialize method, which assigns them to the corresponding variables when it creates the new object. It only does that for this object, not for all future CrazyRobot instances. Those objects will have their own values, depending on the arguments passed to them when Class#new is used..

Bender can't do anything right now, so we need to teach him a method. Let's pretend #bendable? is a method that returns true or false.

class CrazyRobot   def initialize(name, material)     name = name     material = material   end   def bend(object)     if object.bendable?       puts "Shut up, baby, I know it!"       object = "bent #{object}"     else       puts "Bite my shiny, metal ass!     end   end end

Now we can tell our CrazyRobot to bend an object and, if the object is bendable, he will "fix" it:

bender = CrazyRobot.new("Bender Bending Rodriguez", "dolomite (40%)") bender.bend("Good news, everyone")   => "Bite my shiny, metal ass! bender.bend("Professor Farnsworth")   => "Shut up, baby, I know it!"   => "bent Professor Farnsworth"

Unfortunately for the Professor, #bend is a destructive method.

It's highly frowned upon in the bending community, but if Bender decides to self-bend in the privacy of his own home, who are we to judge? Let's create a new class method called self_bend that will take the name variable we created earlier and convert it to "bent (name)".

class CrazyRobot   def initialize(name, material)     name = name     material = material   end   def self_bend     name = "bent #{name}   end end

That should raise an error, because the variable name is local to a different method. For all intents and purposes, it does not exist to our new #self_bend method. To make it accessible to other methods in the object instance, we need to expand its scope by making it an instance variable. Instance variables work just like other variables, but their names start with @ and their scope is the entire object. In other words, all the object's methods can access the instance variable. Here is the code rewritten with instance variables:

class CrazyRobot   def initialize(name, material)     @name = name     material = material   end   def self_bend     @name = "bent #{@name}   end end

You may notice that I didn't make the material variable an instance variable. I kept it as a local variable, because no other methods need access to it as the class is currently written. It's important in writing code that information is only shared when necessary. That way, if a method, for example, doesn't work properly, it narrows down the number of places that you have to look for the problem. It might also be the case that I don't want the user or attackers to be able to access information stored in certain variables, like the password to my CrazyRobot's voice module. There are different ways to hide information and encapsulate parts of a program, but limiting the scope of variables is an important part of that.

Ruby also has class variables, which are shared by all members of the same class. These are not to be confused with instance variables, because unlike instance variables, objects that have the same class variable will also share the same value for that variable. One reason you might use a class variable is to keep track of how many CrazyRobots you've created. Every time you initialize a new CrazyRobot, you increment a counter that is a class variable. Every instance of CrazyRobot will have access to that counter via the shared class variable. Class variables should be avoided except in very special circumstances, since subclasses also share the class variable, which can lead to undesired behavior.
